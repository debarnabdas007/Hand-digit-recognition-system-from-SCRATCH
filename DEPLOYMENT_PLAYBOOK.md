# Deployment Playbook: Hand-Digit Recognition System

**Internal Guide for Deploying the Decoupled Microservice**

This playbook documents the exact steps taken to deploy the production system. As a CSE undergraduate, maintaining this record ensures I can reproduce the deployment process and troubleshoot issues.

## Phase 1: Hugging Face Spaces Backend Deployment

### Context
The backend (FastAPI + PyTorch) needed to be deployed on Hugging Face Spaces with Docker support. Standard Git push was impossible because Hugging Face rejects binary .pth model files without Git LFS, and LFS integration was unreliable.

### Workaround: Direct CLI Upload
We completely bypassed Git and used the Hugging Face CLI to upload the entire backend folder directly to the Space root.

### Step-by-Step Commands
```bash
# 1. Install Hugging Face CLI (if not already installed)
pip install huggingface_hub

# 2. Authenticate with your Hugging Face token
huggingface-cli login

# 3. Upload the backend folder directly to the Space
hf upload backend/ . --repo https://huggingface.co/spaces/debarnab007/digit-vision-backend --private

# Note: The trailing slash after 'backend/' is critical
# This pushes the CONTENTS of backend/ to the Space root, not the folder itself
```

### Verification
- Space URL: https://debarnab007-digit-vision-backend.hf.space
- API Docs: https://debarnab007-digit-vision-backend.hf.space/docs
- Test Endpoint: `GET /` should return health check JSON

### Key Points
- **No Git LFS Required**: CLI upload accepts binary files directly
- **Private Space**: Used `--private` flag for security during development
- **Full Folder Upload**: Includes Dockerfile, requirements.txt, model .pth, and all code
- **Automatic Containerization**: Hugging Face builds Docker image from uploaded files

## Phase 2: Streamlit Community Cloud Frontend Deployment

### Context
The frontend (Streamlit + Canvas) was deployed on Streamlit Community Cloud. Initial attempts failed due to dependency conflicts with Streamlit's internal UI libraries, causing crashes during build.

### Workaround: Aggressive Dependency Stripping
We identified that heavy ML dependencies (torch, rich) were conflicting with Streamlit's internal libraries. The solution was to create a minimal requirements.txt containing only essential packages.

### Modified requirements.txt
**Before (causing crashes):**
```
streamlit
streamlit-drawable-canvas
requests
Pillow
numpy
torch
torchvision
rich
```

**After (working deployment):**
```
streamlit-drawable-canvas
requests
Pillow
numpy
```

### Deployment Process
1. **Connect Repository**: Link GitHub repo to Streamlit Cloud
2. **Set Main File**: `Frontend/ui.py`
3. **Automatic Deployment**: Cloud detects changes and rebuilds on push
4. **URL Generation**: Streamlit provides public URL automatically

### Key Points
- **Removed torch**: Eliminated PyTorch to avoid library conflicts
- **Removed rich**: Removed rich formatting library (used in backend logging)
- **Kept Essentials**: Only UI-specific packages for canvas drawing and HTTP requests
- **Lightning Builds**: Minimal dependencies result in fast, reliable deployments
- **No Docker Required**: Streamlit Cloud handles Python environment automatically

## Troubleshooting Notes

### Backend Issues
- If model fails to load: Verify .pth file is in `models/` folder during CLI upload
- If API returns 500: Check Hugging Face Space logs for PyTorch errors
- If port issues: Hugging Face uses 7860 internally, but FastAPI defaults to 8000

### Frontend Issues
- If build fails: Double-check requirements.txt has no conflicting packages
- If canvas not loading: Ensure streamlit-drawable-canvas is listed
- If API connection fails: Verify backend URL is correct and Space is running

### General
- Always test locally with `docker-compose up --build` before deploying
- Keep deployment branches separate from development
- Document any environment variable changes
